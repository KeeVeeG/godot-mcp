================================================================================
EVALUATE_EXPRESSION CODE VARIANTS
================================================================================
Comprehensive list of GDScript code patterns an LLM might produce when calling
the `evaluate_expression` tool. Each entry includes:
  - The raw expression string (what the LLM sends)
  - Expected result type: VALUE | VOID | ERROR
  - Expected execution path:
      EXPR    = Godot's Expression class (single expression, void-safe)
      ESCRIPT = EditorScript._run() fallback (multi-statement, full editor API)
      REFCNT  = RefCounted.eval() fallback (value-returning with `return`)
  - Why: rationale for categorization

================================================================================
CATEGORY 5: MULTI-STATEMENT SCRIPTS (EDITORSCRIPT ONLY)
================================================================================

Case 25: var m = BoxMesh.new()\nm.size = Vector3(1, 2, 3)\nvar root = edited_scene_root\nvar node = MeshInstance3D.new()\nnode.mesh = m\nroot.add_child(node)\nnode.owner = root
  Result: VOID (returns "ok")
  Path:   ESCRIPT — multi-line script with variable declarations, object
           creation, and multiple operations. The `\n` characters in the
           raw expression are real newlines — _execute_in_editor splits on
           them and indents each line for the _run() wrapper.

Note: This is the most common pattern for complex scene manipulation.
An LLM will try to write a GDScript snippet rather than a single expression.

CATEGORY 13: EDGE CASES & POTENTIALLY PROBLEMATIC INPUTS
================================================================================

Case 50: <empty string>
  Result: ERROR ("expression is required")
  Path:   NONE — caught by the first guard before any compilation attempt

Case 51: <whitespace only: "   ">  
  Result: ERROR — GDScript will fail compilation (not valid expression)
  Path:   Expression.parse() fails → EditorScript.reload() fails →
           RefCounted.reload() fails → {"error": "Failed to compile expression"}

Case 52: pass
  Result: VOID (returns "ok")
  Path:   ESCRIPT — `pass` is a statement, not an expression. Expression class
           won't parse it. EditorScript wraps it as a function body line, works.

Case 53: return 42
  Result: ERROR — Using `return` at top level in EditorScript._run() would
           cause a parse error. RefCounted path prepends another `return`,
           making it `return return 42` which also fails.
  Path:   All paths fail → compilation error

Case 54: var x = 5\nreturn x
  Result: ERROR — Same `return` issue. ESCRIPT can't have return in void _run().
           REFCNT becomes `return var x = 5\nreturn x` which is invalid.
  Path:   All paths fail

Case 55: for i in range(10):\n\tprint(i)
  Result: VOID (returns "ok" via ESCRIPT)
  Path:   ESCRIPT — for-loop is a statement; Expression can't parse it;
           EditorScript wraps it in _run() and it works

Case 56: for node in edited_scene_root.get_children():\n\tif node is MeshInstance3D:\n\t\tnode.visible = false
  Result: VOID (returns "ok" via ESCRIPT)
  Path:   ESCRIPT — for + if is multi-statement, indented correctly by
           the `\n\t` replacement

Case 57: <extremely long expression: 10000+ characters of chained calls>
  Result: Depends on content; likely VALUE or VOID
  Path:   EXPR if it's a single expression, ESCRIPT otherwise
  Why:    LLMs sometimes produce verbose code with excessive chaining

Case 58: # this is a comment\nvar x = 5\nprint(x)
  Result: VOID (returns "ok")
  Path:   ESCRIPT — comments in GDScript start with #; EditorScript handles
           multi-line with comments correctly

Case 59: const PI = 3.14159\nprint(PI * 2)
  Result: VOID (returns "ok")
  Path:   ESCRIPT — const declaration is a statement; works in _run()

Case 60: var result = []\nfor child in edited_scene_root.get_children():\n\tresult.append(child.name)\nprint(result)
  Result: VOID (returns "ok")
  Path:   ESCRIPT — complex script with array building in a loop

CATEGORY 16: ERROR-PRONE PATTERNS (LLM HALLUCINATIONS)
================================================================================

Case 69: get_node("Player")
  Result: ERROR or unexpected value
  Path:   EXPR — base_obj is SceneTree, not a Node. get_node() is a Node
           method. Expression will fail because SceneTree doesn't have
           get_node(). Falls through. EditorScript has no implicit get_node()
           either. REFCNT eval has no scene access.
  Why:    LLM assumes get_node() is globally available like in script context

Case 70: $Player
  Result: ERROR — $ syntax is syntactic sugar for get_node() in GDScript
           but is NOT parseable by Expression class. In EditorScript _run(),
           there's no implicit self to resolve the $ against.
  Path:   EXPR fails to parse → ESCRIPT compiles but fails at runtime
           (no implicit self/owner context for $)

Case 71: get_tree()
  Result: ERROR — In Expression class, get_tree() is a Node method, not
           available on the SceneTree base object. In EditorScript._run(),
           there's no implicit node context.
  Path:   All paths likely fail
  Why:    LLM assumes get_tree() is globally callable

Case 72: set_position(Vector3(1,2,3))
  Result: ERROR — No implicit node context; set_position is a Node method
  Path:   All paths fail

Case 73: load("res://assets/model.glb")
  Result: VALUE (Resource)
  Path:   EXPR may work if load() is available as global; falls to REFCNT
           where `return load(...)` works

Case 74: preload("res://scenes/enemy.tscn")
  Result: VALUE (PackedScene) or ERROR
  Path:   REFCNT via `return preload(...)` — preload may fail at reload time
           depending on context; EXPR won't handle it

Case 75: var inst = preload("res://scenes/enemy.tscn").instantiate()\ninst.position = Vector3(10, 0, 0)\nedited_scene_root.add_child(inst)\ninst.owner = edited_scene_root
  Result: VOID (returns "ok")
  Path:   ESCRIPT — multi-statement instantiation pattern, very common
           LLM pattern for spawning scene instances

================================================================================
CATEGORY 20: PATTERNS THAT REVEAL THE 3-STEP FALLBACK
================================================================================

Case 91: 2
  Result: VALUE (2)
  Path:   EXPR — trivial expression; tells us which path handles single values

Case 92: Vector3(1,0,0).dot(Vector3(0,1,0))
  Result: VALUE (0.0)
  Path:   EXPR — chained method on constructed built-in type

Case 93: var x = 10\nvar y = 20\nx + y
  Result: This is the KEY multi-step tricky case.
           - EXPR: FAILS (var declarations are statements, not expressions)
           - ESCRIPT: WRAPS as `_run() -> void:\n\tvar x = 10\n\tvar y = 20\n\tx + y`
             Compilation succeeds (x + y is a valid expression statement).
             But _run() returns void, so the last line's value is discarded.
             Returns "ok".
           - Does NOT reach REFCNT because ESCRIPT compiled OK.
  Result: VALUE = "ok" (the "ok" string from ESCRIPT, NOT the value 30!)
  Path:   ESCRIPT
  Why:    LLM expects 30, gets "ok". The value of x+y is evaluated but
           discarded because _run() is a void function. This is a design
           limitation: multi-statement scripts that end with an expression
           don't return the expression's value.

Case 94: var root = edited_scene_root\nroot.get_child_count()
  Result: VALUE = "ok" (same issue as Case 93)
  Path:   ESCRIPT — compiles OK as void script; last expression value discarded

Case 95: var node = edited_scene_root.find_child("Player", true, false)\nnode.position
  Result: VALUE = "ok" (expression value discarded by void _run())
  Path:   ESCRIPT
  Why:    Critical: an LLM reading a node property via multi-statement won't
           get the value — they get "ok". The only way to get a value from
           multi-statement code is to avoid var declarations and do it as
           a single expression.


================================================================================
SUMMARY OF EXECUTION PATHS
================================================================================

PATH: Expression class (expr.execute())
  - Handles: single expressions only (no var/for/if/func)
  - Base object: Engine.get_main_loop() (SceneTree)
  - Returns: the expression's result value, or null for void calls
  - Fails fall through to ESCRIPT

PATH: EditorScript._run() (GDScript extends EditorScript)
  - Handles: any valid GDScript that compiles and runs in _run()
  - Always returns "ok" on success (void function)
  - Expression values at end of script are DISCARDED
  - Can access EditorInterface (editor singleton)
  - Has editor plugin context indirectly
  - Falls through to REFCNT if compilation fails

PATH: RefCounted.eval() (GDScript extends RefCounted)
  - Handles: value-returning expressions wrapped with `return`
  - Source: "extends RefCounted\n\nfunc eval():\n\treturn <code>"
  - Returns: the eval result
  - No special context access (no editor, no scene tree)
  - Last resort; catches compilation errors from both above paths
